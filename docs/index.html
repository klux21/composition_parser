composition_parser

Zero-Copy C Parser for the Data Composition Format

klux21/composition_parser is a compact, zero-copy C parser for the Data Composition Format — a deterministic, hierarchical, untyped, whitespace-tolerant configuration and data representation format.

The format is defined in an independent RFC-style specification: Data Composition Format Specification

About the Data Composition Format

The Data Composition Format is designed for human-readable configuration files, lightweight data serialization, small text databases, and text-based protocols. It intentionally avoids type semantics and focuses on deterministic structure.

The Data Composition Format is a very small, hierarchical, untyped, whitespace-tolerant, block-structured text format designed for human-readable configuration files, lightweight data serialization, small text databases, and text-based protocols.

The format does not define data types. All values are strings or blocks containing Data Composition subdocuments.
“Data Composition Format” because it’s exactly that: an untyped format that can hold any kind of data.

Key Properties

  • Untyped — values are raw tokens; interpretation is application-defined.
  • Deterministic grammar — entries, values, blocks, sections, comments.
  • Whitespace-tolerant — whitespace may appear between any tokens.
  • Hierarchical — nested blocks and sections.
  • Suitable for zero-copy parsing — the format is designed for it.
  • Robust under human editing — forgiving syntax, stable structure.
  • Minimal syntax — easy to parse in embedded environments.

Parser Features

  • Zero-copy operation — names, values, and blocks can be returned as slices of the input buffer.
  • No dependencies — implemented in two files: iniparse.c and iniparse.h.
  • Preserves raw content — quoted strings, escapes, and block contents remain untouched.
  • Optional unquoting/unescaping — provided as helper functions.
  • Optional numeric interpretation — via str2num, not required.
  • Section iteration — supports dotted section names and nested sections.
  • Anonymous blocks and entries — supported according to the standard.

What the Format Makes Possible

Below is an example demonstrating mixed styles, nested blocks, numeric formats, dotted sections, and comment types — all in one document. The Data Composition Format enables structured documents that exceed the capabilities of common formats such as INI, JSON, TOML, YAML, XML, and others: 

# You can
# - just iterate over a bunch of numbers, characters, and words like the ones below
1 2 33 42 5 6 7 8 9 0 5 a b c d e f g h i j cat dog horse
_____________________________________
#* - read a bunch of entries that hold more exotic numbers like below 

   (The project contains a speed test that reads all of these as int64_t or
    double in less than a microsecond on a Raspberry Pi 5. And yes, the format
    supports block comments like this one.) *#

inttests = { ib=0b1111 io=0o1234567 id=000056789 ix=0xabcd987 }
floattests = { fb=0b11.11e100 fo=0o1234.56e10 fd=1.2345e64 fx=0xabc.defp10 }
_____________________________________
# - read the points of the triangles and a rectangle for a drawing like this
drawing = { triangle_3D = {{6 4 3}{4 5 7}{-1 17 2}}
            triangle_3D = {{3 2 3}{7 6 2}{1 11 -2}}
            rectangle_3D = {{6 5 6}{8 5 5}{6 15 5}{8 15 5}} }
_____________________________________
# - have a sectionless configuration without useless quotes and commas
server = {
   host = localhost
   port = 8080
   tls = {
      enabled
      certificate = /etc/certs/server.pem

      ciphers = { #* comment block *#
      accept = { TLS_AES_128_CCM_8_SHA256  TLS_CHACHA20_POLY1305_SHA256
                 TLS_AES_128_GCM_SHA256 }
      }
   }
}
_____________________________________

# - and you can also use INI-like configurations, even in the same
#   document, with all the other content above and
# - add sections and entries that share a name as many as you like

Performance

The composition_parser is engineered for extremely high parsing performance. Its deterministic grammar, whitespace-tolerant tokenization, and zero-copy model allow it to process complex hierarchical documents with exceptional efficiency.

Microbenchmark Results

On a Raspberry Pi 5, a tree walk of the 1753 bytes of the document above requires only 3.3 µs to locate:

  • 3 sections
  • 20 blocks
  • 81 entries

A stripped version of the same document (no comments, no blank lines) is parsed in 2.5 µs.

Large Document Benchmark

A tree walk of a converted JSON benchmark file citm_catalog.json of 1673723 bytes containing:

  • 21388 blocks
  • 16390 entries

requires ≈ 2032 µs without generating any output. This corresponds to an effective parsing throughput of about 823 MB/s on a Raspberry Pi 5.

Structural Traversal

The values are the average of 1000 repetitions of a build with gcc version 14.2.0. The measurements represent a pure structural iteration: locating sections, blocks, and entries without performing any semantic processing. No unescaping, unquoting, numeric interpretation, or value transformation is performed during these benchmarks. This demonstrates the inherent efficiency of the Data Composition Format’s grammar.

Because the format is untyped and designed for deterministic scanning, applications may choose between:

  • pure structural traversal (as in the benchmarks)
  • selective unescaping or unquoting only when needed
  • optional numeric interpretation using compiler functions or external libraries

This separation of structure and semantics allows the parser to achieve extremely high throughput while giving applications full control over how values are interpreted.

Benchmark Platform

All performance values were measured on a Raspberry Pi 5 and are the average of 1000 repetitions. This platform is one of the few widely available and sufficiently standardized ARM systems, making its results reproducible and meaningful across different environments. Its predictable thermal behavior and stable clock frequencies allow reliable microsecond-level measurements without interference from dynamic frequency scaling.

A Ryzen 3900X desktop CPU can parse the same 1673723 byte document in 1062 µs (≈ 1.06 ms). Of course desktop-class CPUs vary significantly in microarchitecture, memory subsystem, dynamic clock scaling (especially under Windows), cache hierarchy, mainboard, firmware, thermal behavior, and operating system scheduling. Because of these variations, desktop results do not provide a stable comparison baseline. An unmodified Raspberry Pi 5, however, offers consistent, reproducible performance characteristics.
